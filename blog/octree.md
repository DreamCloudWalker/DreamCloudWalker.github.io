## 场景管理：八叉树 + 视锥体剔除

这个 Demo 在一片空间里散布了几百个方块，用**八叉树（Octree）**把空间递归切分，再用**视锥体剔除（Frustum Culling）**只画出主相机看得见的物体。

- **左窗口（主相机）**：就是用来剔除的那个相机，只画通过剔除的物体。
- **右窗口（上帝视角）**：俯瞰全局——画出八叉树的节点盒子（线框）、所有物体（**绿色/彩色=被保留绘制，灰色=被剔除**），方便你直观看到剔除发生。

用 **WASD 平移**、**拖动右窗口旋转**主相机，观察灰色（被剔除）和彩色（被绘制）的物体随视锥转动而实时变化。面板上的"绘制物体 X / 总数"会告诉你省了多少。

### 一、为什么需要场景管理

一个大场景里可能有成千上万个物体，但相机一次只能看到其中一小部分。如果每帧都把所有物体都提交给 GPU，会浪费大量时间在**根本不在屏幕上**的物体上。

**视锥体剔除**：相机的可视范围是一个由 6 个平面围成的"视锥体（frustum）"。只有落在这个锥体内的物体才可能被看到，锥体外的可以直接跳过。

问题来了：逐个测试每个物体"在不在视锥里"，本身就是 O(N) 的开销。物体越多越慢。这就需要**空间分割结构**来加速。

### 二、八叉树：把空间递归切成 8 份

**八叉树**把一个立方体空间沿 x/y/z 三个轴各切一刀，分成 **2×2×2 = 8 个子立方体**；每个子立方体如果还装着很多物体，就继续往下切，直到：

- 节点里的物体数足够少（本 Demo ≤ 6 个），或
- 达到最大深度（本 Demo 4 层）。

```
        根节点(整个空间)
        /  /  | |  \  \  \  \
      8 个子立方体（继续递归细分…）
```

物体按它的中心点归属到对应的子节点。这样空间上邻近的物体被组织进了树的同一棵子树里——**这正是加速剔除的前提**。

### 三、用八叉树加速剔除

剔除时不再逐个测物体，而是**自顶向下遍历八叉树**：

1. 取出相机视锥体的 6 个平面；
2. 测试当前节点的**包围盒（AABB）**与视锥体的关系：
   - **整个盒子在视锥外** → 这个节点连同它下面**所有子节点、所有物体**一次性跳过，不再深入；
   - **盒子与视锥相交或在内** → 递归测试它的 8 个子节点，并绘制本节点直接挂着的物体。

关键在第一种情况：一次盒子测试，就剔掉了一大片物体。比如根节点的某个角落整体在视野外，那一整棵子树（可能上百个物体）一次就被排除了。复杂度从 O(N) 降到接近 **O(log N + 可见数)**。

> 关掉「启用视锥剔除」：所有物体无条件全画。对比面板上的"绘制物体"数量和帧率，能看出剔除省下了多少。右窗口也会看到原本灰色（被剔除）的物体全部变回彩色。

### 四、AABB 与视锥体的相交测试

视锥体的每个平面用 `ax + by + cz + d = 0` 表示，法线指向锥体**内侧**。对一个轴对齐包围盒（AABB），用 **p-vertex（正顶点）** 技巧快速判断：

```
对每个平面：
  取 AABB 在该平面法线方向上"最靠正侧"的那个角点 p
  若 p 都在平面外侧(代入 < 0) → 整个盒子必在该平面外 → 盒子在视锥外，剔除
若 6 个平面都没把盒子判为"完全在外" → 盒子可能可见，保留
```

只需比较每个轴的 min/max，挑出正确的角点，6 个平面各算一次点积即可，非常快。

> 视锥体的 6 个平面可以直接从 **view-projection 矩阵**里提取（Gribb–Hartmann 方法）：把矩阵的行做加减组合就得到各平面系数，再归一化。本 Demo 用的就是这种方法。

### 五、四叉树？八叉树？

- **八叉树（Octree）**：三个维度都切，适合**真正的 3D 场景**（物体在高度上也分布很广，如本 Demo）。
- **四叉树（Quadtree）**：只切 x/z 两个水平维度，适合**地形、俯视类**场景（高度差不大），更省内存。

二者思想相同，只是切分维度不同。地形 chunk 管理常用四叉树，密集 3D 物体管理常用八叉树。

### 小结

- 场景管理的目标：**只处理可能被看见的物体**，跳过视野外的；
- **视锥体剔除**判断物体是否在相机的 6 面锥体内；
- **八叉树**把空间递归 8 分，让"一次盒子测试剔掉一整片物体"成为可能，把剔除从 O(N) 降到接近 O(log N)；
- AABB-视锥相交用 p-vertex 技巧 + 从 VP 矩阵提取的 6 个平面，又快又准；
- 俯视类用四叉树、密集 3D 用八叉树，思想一致。
